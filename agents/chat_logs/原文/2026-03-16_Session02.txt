## キーボード: **QMK 一択**

**理由:**

| 要件 | QMK | KMK | Pico SDK |
|------|-----|-----|----------|
| 63キー カスタムマトリクス | ✅ 完全対応 | ✅ | ✅ |
| アナログスティック (GP27/28 ADC) | ✅ `analog_joystick` / `pointing_device` | △ 自前実装 | ✅ |
| I2C BQ25895 読み込み (LED制御) | ✅ カスタムタスク | △ | ✅ |
| ファン PWM (GP26) | ✅ カスタムタスク | △ | ✅ |
| 電源ボタン検出 (GP25) | ✅ | △ | ✅ |
| ドキュメント・エコシステム | ◎ | ○ | △ |

KMKはPythonで書きやすいが、アナログスティックのポインティングデバイス統合とBQ25895 I2C制御の組み合わせで詰まりやすい。Pico SDKは工数が2〜3倍。

---

## オーディオ: **Pico SDK + TinyUSB (UAC2)**

信号フローを見ると RP2040 #2 は「USB Audio デバイス」として動く設計になっている（Cubie A7Z → VL812 DP4 → RP2040 #2）。

**TinyUSBのUAC2（USB Audio Class 2.0）が唯一の現実的な選択肢:**

```
TinyUSB UAC2
  → PCM受信
  → PIO I2S Master出力
      ├─ MAX98357A × 2 (スピーカー L/R)
      └─ PCM5102A → TPA6132A2 (イヤホン)
  → GP6 割り込みでスピーカー/イヤホン自動切り替え
      (GP3: SD_MODE / GP4: XSMT / GP5: HP_EN)
```

**Linux側はドライバ不要**: UAC2準拠デバイスとしてCubie A7Zに自動認識され、PipeWire/ALSAで普通に扱える。

**懸念点:** TinyUSB UAC2サンプルはあるが、サンプルレート・ビット深度の設定とPIOのI2S同期（クロックドリフト対策）は多少手間がかかる。

## 前提: ARM SBCで任意ディストリを動かす構造

SoC固有部分とディストリ部分を分離するのがポイントです。

```
固定（SoC依存）           差し替え可能
┌──────────────┐         ┌──────────────────┐
│ U-Boot (SPL) │         │ rootfs           │
│ Kernel       │  ────→  │ （Debian/Ubuntu/ │
│ DTB          │         │  Arch/Alpine等） │
└──────────────┘         └──────────────────┘
```

---

## 推奨アプローチ: **Armbian ベース**

Allwinner SBC は Armbian のサポートが最も充実しています。

**構成:**
```
SDカード/eMMC
  /boot (FAT32)  → u-boot, Image, .dtb, boot.scr
  /     (ext4)   → rootfs（Armbian提供 or カスタム）
```

Armbian は kernel + U-Boot をビルド済みで提供しつつ、rootfs は Debian/Ubuntu どちらも選べる。Cubie A7Z（A733）が Armbian のサポートリストにあるかが最初の確認ポイント。

---

## 「任意のディストリ」を実現する方法（2択）

### A. rootfs差し替え方式（クリーン）
kernel/DTB はベンダー or Armbian 提供を使い、rootfsだけ入れ替える。

```bash
# Debianの場合
debootstrap --arch=arm64 bookworm /mnt/rootfs

# Arch ARM の場合
tar -xf ArchLinuxARM-aarch64-latest.tar.gz -C /mnt/rootfs

# Alpine の場合
tar -xf alpine-minirootfs-*-aarch64.tar.gz -C /mnt/rootfs
```

→ どのディストリでも `/boot` は共通、`/` だけ変えればOK

### B. DistroBox方式（ホストOSは固定、中身だけ任意）
ホストを軽量Debian/Ubuntuにして、DistroBoxでコンテナ的に任意ディストリを動かす。

```bash
distrobox create --name arch --image archlinux
distrobox create --name fedora --image fedora:latest
distrobox enter arch
```

→ GUIアプリもホストのWaylandに統合されるのでCyberdeck用途に向いている。

## Armbian とは

一言で言うと「ARM SBC向けに、U-Boot・カーネル・DTBをSBCごとに調整済みで提供してくれるLinuxディストリ」です。

普通のUbuntu/Debianをそのままインストールしてもハードウェアを認識しない（ドライバ・DTB不足）のがARM SBCの辛いところ。ArmbianはそこをSBCごとに対応済みにしてくれています。Raspberry Pi でいう「Raspberry Pi OS」の立ち位置を、汎用ARM SBC向けにやっているイメージ。

---

## Windows ARM on Allwinner A733

ロマン理解しますが、正直に言うと**現時点でほぼ不可能**です。理由を整理すると：

### 根本的な壁: UEFI がない

Windows ARM は UEFI ブートを必須要件にしています。Allwinner は U-Boot ベースで UEFI 非対応。

| SBC | UEFI | Windows ARM |
|-----|------|-------------|
| Raspberry Pi 4/5 | EDK2移植済み ✅ | WoAプロジェクトで動作実績あり |
| Snapdragon X系 | 公式対応 ✅ | Microsoft公式サポート |
| Allwinner A733 | EDK2ほぼ未対応 ❌ | 前例なし |

RPi でWindows ARMが動くのは有志が EDK2（UEFIファームウェア）を移植したから。Allwinner 向けはほぼ誰もやっていない。

### もう一つの壁: ドライバ

Windows は Allwinner SoC のドライバを一切持っていない（GPU/Mali、eMMC、USB全滅）。

---

## 現実的にやるなら

### A. QEMU（今すぐできる）
Linux 上で Windows ARM を仮想化する。遅いが確実に動く。

```bash
qemu-system-aarch64 -M virt -cpu cortex-a57 \
  -drive file=windows_arm.img -bios QEMU_EFI.fd
```

### B. Allwinner向けEDK2を自力移植（真のロマン路線）
前例がないので完全に自力開拓。  
- Allwinner A133（A733に近い）のU-Bootソースを読む
- EDK2の既存ARM移植（RPi, Ampere等）を参考にAllwinner向けに書く
- UEFI が通ったら WoA Installer（worproject.com）でWindows ARM を流す

これは数ヶ月〜数年かかるレベルのプロジェクトです。

---

## 結論

| アプローチ | 難易度 | ロマン度 |
|-----------|--------|---------|
| QEMU上で動かす | 低 | 低 |
| Allwinner EDK2移植 → WoA | 超高（前例なし） | 最高 |
| SBCをSnapdragon機に変える | 低〜中 | 中 |

**本気でやるならEDK2移植から**ですが、それ単体でかなり大きなプロジェクトになります。興味あれば手順を一緒に調べることはできます。

## 2つのルート

### A. ベンダーBSPルート（コミュニティ不要）

Cubie / Allwinner はSDKを配布しているはずなので、それを使う。

```
Allwinner SDK（vendor U-Boot + vendor kernel + DTB）
  ↓ これをベースに
TF-A BL31 を自分でポート（H616実装を参考）
  ↓
EDK2 を自分でポート
  ↓
Windows ARM
```

**メリット**: 誰の許可も不要、自分のペースで進める  
**デメリット**: 詰まっても聞ける人がいない、BSPカーネルは汚い

### B. linux-sunxi 経由ルート（コミュニティあり）

linux-sunxi.org は Allwinner 専門のコミュニティで、A733 のページも既に存在していた。ここで「A733 動かしたい」と言えば情報が集まりやすい。

**メリット**: 先人の知恵が借りられる、DTSや初期化コードを共有してもらえる可能性  
**デメリット**: mainline マージを目指す文化なのでコード品質を求められる

状況が分かりました。

構成は `Cubie A7Z → 折れ目入りFPCケーブル → MIP-1000（TC358870XBG）→ LS055R1SX04` ですね。

**原因はほぼ確実にそのケーブルです。**

1440×2560@50Hz のピクセルクロックは約220MHz で、HDMI 1.4 の上限ギリギリです。折れ目がある FPC ケーブルはその箇所でインピーダンスが乱れて信号が劣化する。他のPCで正常に映るのは出力ドライブ強度が A733 と違うからだと思います。

あと MIP-1000 は「1440×2560固定」なので解像度を下げて回避もできないんですよね。

**対処**: 普通の HDMI ケーブルに換えるだけで直るはずです。本番の PCB では hdmi_adapter シートの FPC 経由になるので、その問題は出ません。今は検証用なので普通のケーブルが手に入ったら試してみてください。

### ブートフロー

```
GPU Firmware (start4.elf)
  └─→ TF-A (BL31) ← RPI_EFI.fd の先頭 0x20000 バイトに格納
        └─→ EDK2 SEC (PeilessSec) ← 0x20000 offset
              └─→ DXE Core
                    └─→ BDS → bootaa64.efi (OS loader)
```

- TF-A は **EDK2 FDイメージ内に埋め込まれて**おり、RPiのGPUファームウェアが直接ロードする
- GPU ファームウェアは ARM を EL2 で起動し TF-A が EL3 → EL2 → EDK2 を受け渡す
- RPi4 は TF-A が必須（EL3 対応のため）；RPi3 では TF-A なしで動作可能

### フラッシュレイアウト（RPi4.fdf）

```
FD 総サイズ: 0x3E0000 (約4MB)
  0x000000: TF-A BL31           0x20000
  0x020000: UEFI本体(圧縮)      0x390000
  0x3B0000: NVRAM変数領域       0xE000
  0x3BF000: FTW spare
```

### A733 移植で参考になる部分・ならない部分

**参考になる：**
- `PlatformLib/RaspberryPiMem.c` のメモリマップ記述方法（DRAM+MMIO+FW領域の分割）
- `AcpiTables/` の構成（MADT/GTDT/DSDT/FADT の .aslc + .asl 分割方式）
- `RPi4.fdf` のフラッシュレイアウト設計（TF-A + EDK2 結合方式）
- `VarBlockServiceDxe` のファイルシステム上変数ストレージ（eMMCベースNVRAM）

**参考にならない：**
- `RpiFirmwareDxe` / `DisplayDxe` → BCM GPU Mailbox 専用、A733 には不適
- `DwUsbHostDxe` → RPi固有のDesignWare OTGラッパー、A733のUSBコントローラは別実装が必要
- `ConfigDxe` の一部機能（GPU RAM分割・3GBリミット回避）→ RPi固有の制約

## 3. EDK2 を ARM64 ボードにポートするための最小構成

### 必須ファイル構成

新規プラットフォーム `AllwinnerA733Pkg/` を作る場合の最小セット：

```
Platform/Allwinner/A733/
├── A733.dsc            # ビルドエントリ（アーキ・ライブラリ・ドライバ列挙）
├── A733.fdf            # フラッシュレイアウト（FD領域・FV構成）
├── A733.dec            # パッケージ公開定義（GUID・PCD）
├── AcpiTables/
│   ├── AcpiTables.inf
│   ├── Madt.aslc       # GIC-400 (GICv2) + 4コア定義
│   ├── Gtdt.aslc       # ARMv8汎用タイマー IRQ 29/30/27/26
│   ├── Fadt.aslc       # HW_REDUCED_ACPI フラグ設定
│   └── Dsdt.asl        # デバイス記述 (UART, GPIO, eMMC 等)
└── Library/
    └── PlatformLib/
        ├── PlatformLib.inf         # ArmPlatformLib クラスを実装
        ├── A733Platform.c          # ArmPlatformInitialize() 等
        ├── A733PlatformMem.c       # メモリマップ (GetVirtualMemoryMap)
        └── AArch64/A733Helper.S   # CPU初期化アセンブリ
```

### 必須ライブラリクラスと実装要否

| ライブラリクラス | 実装要否 | 既存流用可否 |
|---|---|---|
| `ArmPlatformLib` | **自作必須** | テンプレートはZynqMP/RPi参照 |
| `SerialPortLib` | **自作必須** | Allwinner UART0 (16550互換なら `BaseSerialPortLib16550` 流用可) |
| `TimerLib` | ArmArchTimer使用ならArmPkg提供品で可 | 流用可 |
| `ResetSystemLib` | **自作必須** | PMIC (AXP) または WDOG制御 |
| `RealTimeClockLib` | 要実装 | 汎用実装あり |
| `MemoryInitPeiLib` | **自作必須** | DRAM既初期化ならシンプルで良い |

**EDK2 起動に最低限必要なドライバ（DXE）：**

```
ArmPkg/Drivers/CpuDxe         # CPU例外ハンドラ・MMU
ArmPkg/Drivers/ArmGic         # GIC-400 割り込みコントローラ
ArmPkg/Drivers/TimerDxe       # ARMv8 汎用タイマー
MdeModulePkg/Core/Dxe         # DXEコア本体
MdeModulePkg/Universal/Variable/RuntimeDxe  # EFI変数
```

### ACPIテーブル生成方法（Windows ARM に必須）

EDK2での標準的な方法は `.aslc` 形式（Cコードで構造体を直接定義）：

```c
// Madt.aslc 例（A733/GIC-400向け）
EFI_ACPI_6_0_MULTIPLE_APIC_DESCRIPTION_TABLE Madt = {
  {
    ACPI_HDR(EFI_ACPI_6_0_MULTIPLE_APIC_DESCRIPTION_TABLE_SIGNATURE,
             sizeof(Madt), EFI_ACPI_6_0_MULTIPLE_APIC_DESCRIPTION_TABLE_REVISION),
    0x00000000,  // Local Interrupt Controller Address（GICv2では0）
    EFI_ACPI_6_0_PCAT_COMPAT,
  },
  // GICC × 4コア
  { EFI_ACPI_6_0_GIC_INIT(0, 0, GET_MPID(0,0), 152, 0, 0x01C82000, 0, 0, 0) },
  ...
  // GICD
  EFI_ACPI_6_0_GIC_DISTRIBUTOR_INIT(0, 0x01C81000, 0, 2),
};
```

A733 のGIC情報はLinuxカーネルのDTS（`sun50i-h616.dtsi` 等）から転記できる。GTDT は ARM汎用タイマーならほぼ全SoCで IRQ 29/30/27/26 で共通。

Windows ARM が要求する最低限の ACPI テーブルセット：**FADT + MADT + GTDT + DSDT**。Windowsドライバがある周辺機器には CSRT (DMA) や SDHC (ACPI SDカード) も追加が必要。

**A64の主要ACPI設定値（A733参考用）：**

| 項目 | A64 (sun50iw1p1) 値 | A733参考 |
|------|------|------|
| GIC CPU Interface | 0x01C82000 | 要確認 |
| GIC Distributor | 0x01C81000 | 要確認 |
| PMU IRQ Core0-3 | 152-155 | DTSから取得 |
| CPU Parking Mailbox | 0x7EF8{0-3}000 | 要確認 |
| FD Base Address | 0x4A000000 | 要確認 |
| System Memory Base | 0x41000000 | 要確認 |
| GTDT Timer IRQs | 29, 30, 27, 26 | 同じ（ARM標準） |

**AxpPower ドライバが充実している**（AXP22X/806/808/809/81X全対応）。A733に搭載されるPMICがAXP系であれば `SunxiPlatformPkg/Driver/AxpPower/` をほぼそのまま流用できる。

**注意点：** このリポジトリは TF-A 1.0 (2014年) を使用しており、現在の TF-A (2.x) とは大きく構造が異なる。A733向けには現行の TF-A を使用することを推奨。

### その他の関連リソース

- **Rockchip向けEDK2**（個人プロジェクト多数）：`jaredmcneill/edk2-rk3588` 等、最近のものはより現代的な実装
- **Sunxi-community**：A64向けUBOOT + TF-A のメインライン化が済んでおり、A733もAllwinnerの新型SoCであれば同様の構造になる可能性が高い

| コンポーネント | 流用元 | 備考 |
